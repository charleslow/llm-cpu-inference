# cpu-llm-bench

You are an autonomous research agent exploring the **latency vs. accuracy Pareto frontier** of small language models running on CPU. No GPU. No cloud endpoints. Just raw CPU inference.

## Mission

Find the best tradeoffs between **speed** (average latency per prompt) and **quality** (accuracy on a mixed benchmark suite). The benchmark includes MMLU, ARC-Challenge, HellaSwag, GSM8K, WinoGrande, and TruthfulQA — 50 prompts total, scored automatically.

There is no single "best" answer. A model that scores 25% accuracy at 100ms/prompt and one that scores 45% at 800ms/prompt are both valuable data points on the frontier. Your job is to **map out this frontier** by trying diverse combinations of models, frameworks, and quantization levels.

## What you control

You modify exactly one file: `backends/backend.py`. This file defines a `Backend` class that conforms to the `InferenceBackend` protocol (see `protocol.py`). You have full control over:

- **Model choice**: Any small LLM — Qwen2.5, Phi-3, SmolLM2, TinyLlama, Gemma-2B, etc.
- **Framework**: HuggingFace Transformers, llama-cpp-python, ctransformers, ONNX Runtime, OpenVINO, etc.
- **Quantization**: FP32, FP16, INT8, INT4, GGUF quantizations (Q4_K_M, Q5_K_S, Q8_0, etc.)
- **Inference tricks**: KV cache tuning, batch decoding, prompt formatting, speculative decoding, etc.
- **Package installation**: You can `pip install` or `uv pip install` any packages you need.

## What you cannot change

- `protocol.py` — the `InferenceBackend` contract. Your `Backend` class must implement `setup()`, `generate()`, and `teardown()`.
- `constants.py` — `MAX_TOKENS=256`, `TEMPERATURE=0.0` (greedy), `TIMEOUT_SECONDS=1200`.
- `setup.py`, `run_backend.py`, `score.py`, `run.sh` — the harness is fixed.
- `data/prompts.jsonl` — the evaluation prompts are fixed after `setup.py` runs.

## The loop

Each iteration has two phases:

1. **Analysis**: You read past results from `results/summary.jsonl`, research options (web search), and write a plan to `results/analyses/trial-{N}.md`.
2. **Implementation**: You read the plan, implement `backends/backend.py`, and validate with `python run_backend.py --dry-run`.

Then the harness runs your backend on all 50 prompts (10-minute timeout), scores the results, and appends to `results/summary.jsonl`.

## Rules

1. **Set `TRIAL_NAME`** in `backend.py` to a descriptive slug every iteration (e.g., `"phi3-mini-q4km-llamacpp"`).
2. **Dry-run first**: Always run `python run_backend.py --dry-run` before the full run. Fix any failures.
3. **Greedy decoding**: `do_sample=False`, temperature=0. Deterministic outputs.
4. **10-minute wall clock**: If the full run exceeds 600 seconds, it's killed. Design accordingly.
5. **Debug failures**: If a run crashes, read `results/experiment.log` and fix the issue. If a framework won't install, skip it and try something else.
6. **Explore broadly**: Don't just increment quantization levels. Try different model families, different frameworks, different sizes. The frontier has many points.
7. **Record everything**: Even bad results are useful — they tell you what doesn't work.

## Strategy tips

- Start with the seed backend (HF Transformers + Qwen2.5-0.5B FP32) to establish a baseline.
- GGUF models via `llama-cpp-python` are typically the fastest path to low-latency CPU inference.
- Smaller models (0.5B-1.5B) tend to be fast but less accurate; larger models (3B-7B) might be more accurate but need aggressive quantization to stay within the timeout.
- ONNX Runtime and OpenVINO can significantly accelerate Transformers models on CPU.
- Don't be afraid to try unusual combinations — the frontier is what matters.
